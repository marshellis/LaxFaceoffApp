# Build & test on a real device (iOS simulator / Android emulator)

This app uses native modules (`react-native-audio-api`, `react-native-mmkv`, `react-native-reanimated`),
so it runs on a **development build** (expo-dev-client), **not Expo Go**. That's expected and is the
modern Expo workflow — and it's exactly what fixes the old "couldn't test custom native audio" pain.

The mobile scripts (`scripts/mobile/`) auto-discover your JDK and Android SDK — including a Homebrew
keg-only `openjdk@17` that isn't on PATH. Run `scripts/mobile/doctor.sh` any time to check.

## This machine's resolved toolchain (Jun 2026)
- **JAVA_HOME** = `/opt/homebrew/opt/openjdk@17/libexec/openjdk.jdk/Contents/Home` (OpenJDK 17.0.19, keg-only)
- **ANDROID_HOME** = `/opt/homebrew/share/android-commandlinetools`
- Maestro 2.5.1, emulator `ACE_Pixel_API_34`, 2 iOS runtimes — all present.
- `scripts/mobile/_env.sh` exports the two `*_HOME` vars; the other scripts source it.

## iOS (simulator) — fastest on a Mac, no extra SDK downloads
```bash
npx expo run:ios               # prebuilds ios/, builds, installs to a booted simulator
scripts/mobile/run-flows.sh ios   # runs the Maestro flows + saves screenshots
```

## Android (emulator) — one-time SDK top-up first
The installed SDK has build-tools 34 + platform-tools + `platforms;android-34`, but a RN 0.81
native build also needs a newer platform + the NDK + CMake. Install them once:
```bash
. scripts/mobile/_env.sh
yes | sdkmanager "platforms;android-35" "ndk;27.1.12297006" "cmake;3.22.1"
# (exact ndk version: match `ndkVersion` in android/build.gradle after the first prebuild;
#  if Gradle complains about a different NDK version, install that one.)
```
Then build + test:
```bash
npx expo run:android                  # prebuilds android/, gradle build, installs to the emulator
scripts/mobile/run-flows.sh android   # runs the Maestro flows + saves screenshots
```

## What the Maestro run proves
`maestro/flows/smoke.yaml` launches the app and asserts the Home screen renders. As more flows are
added (`practice-happy-path.yaml`, etc.) it will tap **START** and assert the running state — the
visual half of the on-device check. The **audio** half (confirming the whistle timing is identical
between a debug dev-build and a release build) is a human listen-test: build both
(`npx expo run:android` debug vs `eas build --profile production`), run a practice set on each, and
confirm the cue spacing matches. The engine schedules cues on the hardware audio clock
(`AudioContext.currentTime`), so they should be identical — this is the fix for the original bug.

## Notes
- `android/` and `ios/` are gitignored — they're generated by `expo prebuild` (Continuous Native
  Generation). Don't commit them; regenerate any time with `npx expo prebuild`.
- If `aapt` is missing during an Android build: `ln -sf "$ANDROID_HOME/build-tools/34.0.0/aapt" /opt/homebrew/bin/aapt`.
